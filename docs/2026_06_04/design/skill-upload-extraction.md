# Design: `.skill` Upload Extraction (Web App)

**Date:** 2026-06-04
**Repo:** `agentic-toolkit-web`
**Status:** Implemented (2026-06-04)

## Implementation notes

- **Binary-safe `FileEntry`** lives in `src/lib/file-entry.ts` (`encoding?: 'utf8' | 'base64'`,
  with `bytesToFileEntry`/`fileToFileEntry` sniffing UTF-8 vs binary, plus shared `basename` and a
  generic `stripCommonRoot`). `Contribute.tsx` and `publish-service.ts` now import these instead of
  keeping their own copies; `publish-service` passes `base64` blobs through verbatim and only encodes
  `utf8`. Read paths (`readFileList`, `walkFsEntry`) read `arrayBuffer` and sniff, fixing the latent
  folder-upload binary corruption bug.
- **Extraction** lives in `src/lib/skill-archive.ts` (`extractSkillArchive`, `isSkillArchive`,
  `hasSkillExtension`, `sanitizeEntryPath`). Wired into `StepFiles` for `type === 'skill'` only,
  routing a single dropped/selected `.skill`/`.zip` to extraction.
- **Frontmatter pre-fill** uses a hand-rolled parser in `src/lib/frontmatter.ts` rather than adding a
  dependency — it handles block scalars (`>`/`|`), the one structure a regex can't, scoped to the
  top-level scalars (`name`/`description`) the wizard needs.

### Resolved open questions

- **Binary detection:** content sniff (strict UTF-8 decode + NUL check), not an extension list — no
  misclassification of unknown extensions.
- **YAML:** hand-rolled `frontmatter.ts`, no `js-yaml` dependency.
- **Size limits:** per-file 10 MB, total 25 MB, max 500 files (constants in `skill-archive.ts`).
- **Zip-slip:** `..`/drive-letter/absolute paths rejected; JSZip already resolves `..` on load, so the
  guard is defense-in-depth and unit-tested directly via the exported `sanitizeEntryPath`.

## Problem

Non-technical users export skills from the Claude desktop app as a `.skill` file. A
`.skill` is a **zip archive** (magic bytes `PK\x03\x04`) whose contents are the raw skill
files nested under a single folder named after the skill:

```
emergent-qa-refinement.skill   (zip)
└── emergent-qa-refinement/
    └── SKILL.md                (YAML frontmatter: name, description)
```

These users publish through the web app's `/contribute` wizard. We do **not** want the
opaque `.skill` archive landing in the registry, because the registry is a git repo of
plain files with PR review, per-file `files[]` lists, and per-file SHA-256 checksums
(`scripts/lib/checksum.ts`). The CLI installer fetches files individually and has no zip
awareness. A binary `.skill` blob would defeat diffing, review, checksums, and install.

**Decision:** Extract the `.skill` on upload, store the raw files (canonical form), and
keep `.skill` entirely a web-app concern. This mirrors how npm stores the unpacked tree,
not the uploaded tarball.

## Scope (this phase)

- IN: Detect a dropped `.skill` (or zip) in the contribute wizard, extract it client-side,
  normalize to `FileEntry[]`, and let the existing publish flow run.
- IN: Make the upload → git-blob pipeline **binary-safe** (skills can bundle images/fonts).
  This also fixes a latent corruption bug on the existing "Add folder" path.
- IN: Pre-fill the wizard's `name` + `description` from `SKILL.md` frontmatter.
- IN: Zip-slip sanitization; scope extraction to `type=skill`.
- OUT (fast-follow): "Download as `.skill`" round-trip dropdown on the asset detail page.

## Findings from the two sample exports

| Aspect | Finding | Design impact |
|---|---|---|
| Container | Standard zip, `PK\x03\x04` | Detect by `.skill` extension; optionally sniff magic bytes |
| Internal layout | Single root folder named after skill (`name/SKILL.md`) | Existing `stripCommonRoot()` handles it as-is |
| Entrypoint | `SKILL.md` (uppercase) | Matches registry/CLI; no casing fixup needed |
| Metadata | YAML frontmatter `name` (kebab-case) + `description` (folded `>` scalar) | Pre-fill wizard; needs a real YAML parse, not regex |
| `manifest.json` / `README.md` | Absent inside the archive | Manifest generated by publish flow; README starts empty |
| Binary assets | None in these samples (SKILL.md only) | Still build binary-safe path for real-world skills |

## Current pipeline (relevant pieces)

- `src/routes/Contribute.tsx`
  - `interface FileEntry { content: string; path: string; size: number }` (~L45) — content is
    UTF-8 **string**; this is the binary-safety blocker.
  - `readDropEntries()` (~L526), `readFileList()` (~L541), `walkFsEntry()` (~L1160) — three
    read paths, all read as text.
  - `stripCommonRoot()` (~L1064) — strips a shared parent prefix (reused for extraction).
  - manifest auto-detect (~L450), README auto-detect (~L473).
  - `StepFiles()` (~L552) — dropzone UI + review listing.
- `src/lib/publish-service.ts`
  - base64-encodes each file's string content → git blob (~L92). Branch point for binary.
- `package.json` — `jszip@^3.10.1` present; **no YAML parser**.

## Changes

### 1. Binary-safe `FileEntry` (backbone)

Extend `FileEntry` to carry an encoding distinction instead of assuming UTF-8 text:

```ts
interface FileEntry {
  path: string;
  size: number;
  // text content kept for preview/editor/auto-detect paths;
  // binary carried as base64 (or bytes) so it round-trips intact.
  encoding: 'utf8' | 'base64';
  content: string; // utf8 text, or base64 when encoding === 'base64'
}
```

- **Read paths**: detect binary (by extension all-list/denylist or content sniff) and read
  binary entries via `readAsArrayBuffer` → base64; keep text as today.
- **`publish-service.ts`**: when creating blobs, pass `content` straight through for
  `base64` entries; utf8→base64 for text entries (current behavior).
- **Review step**: render `(binary, <size>)` for binary entries instead of dumping bytes.
- **Editor/preview/auto-detect**: only operate on `utf8` entries (README editor, manifest
  detection, markdown preview).

> This change is independent of `.skill` and fixes binary corruption on the existing
> folder-upload path too.

### 2. `.skill` extraction source

New helper (e.g. `src/lib/skill-archive.ts`):

- Input: a `File` whose name ends in `.skill` (or whose first bytes are `PK\x03\x04`).
- `JSZip.loadAsync(file)` → iterate entries:
  - Skip directory entries.
  - **Zip-slip guard**: reject any entry whose normalized path escapes root (contains
    `..` segments or is absolute).
  - Per entry: text files → `.async('string')`; binary → `.async('base64')`, set encoding.
- Run the resulting paths through `stripCommonRoot()` to drop the `name/` prefix.
- Return `FileEntry[]` and feed the same state the folder/file inputs feed.

Wire detection into the dropzone (`StepFiles`) and the file `<input>` handlers: if a single
`.skill`/zip is provided, route to extraction; otherwise current behavior. Only active when
`type === 'skill'`.

### 3. Frontmatter pre-fill

- Add a YAML parse for the entrypoint's frontmatter. Options: add `js-yaml` (robust, handles
  folded `>` scalars), or a minimal hand-rolled parser limited to `name`/`description`.
  Recommend `js-yaml` since the folded scalar rules out a naive regex.
- After extraction, if no `manifest.json` is present, parse `SKILL.md` frontmatter and
  pre-fill `name` (kebab-case, already conformant) and `description`. User can edit.

### 4. Safety & scoping

- Zip-slip sanitization (above).
- Reasonable per-file and total-size guards before extraction (client-side; protects the
  user's browser and avoids oversized PRs).
- Extraction + the future `.skill` download are **skill-only**; other asset types never have
  a `.skill` form.

## Edge cases

- `.skill` containing only `SKILL.md` (both samples): works; README step starts empty.
- Root-level `SKILL.md` with no wrapping folder: `stripCommonRoot()` is a no-op — fine.
- Lowercase `skill.md`: normalize entrypoint to `SKILL.md` on extraction (defensive; not
  seen in samples).
- Non-`.skill` `.zip` dropped: treat identically if magic bytes match; otherwise reject with
  a clear message.
- Binary entry on a non-binary-aware branch: guarded by the encoding flag.

## Fast-follow (separate doc): Download as `.skill`

The current download emits `{name}-{version}.zip` containing `manifest.json` + a
`dependencies/` folder — **not** a Claude-desktop-compatible `.skill`. A `.skill` export is
just the skill's own files (`SKILL.md` + `assets/`/`references/`), no `manifest.json`, no
dependencies, nested under a `{name}/` folder, extension `.skill`. Implement as a separate
packaging path behind a download dropdown ("Download" / "Download as .skill"), skill-only.

## Open questions

- Binary detection strategy: extension allow/deny list vs. content sniff vs. both.
- Add `js-yaml` dependency vs. minimal in-repo frontmatter parser.
- Size limits: pick concrete per-file / total caps.
